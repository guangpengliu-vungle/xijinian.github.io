Bean 的装配

自动化装配 Bean

流程

  • 用一个类实现一个接口，并在此类上面标注 @Component 注解
  • 构造一个 Config 类，在上面标注 @Configuration 和 @ComponentScan 注解，意图让 Spring 可以发现此类同包下的 Component 注解
  • 在其他类中，可以直接在最开始定义的接口上标注一个 @Autowired 注解，这样就会把此注解的实现类注入到当前的类中了。

tips

  • 一个类在使用 Component 注解时直接为自己添加一个名字。语法为：

    xxxxxxxxxx
    @Component("xiaoHong")
    public class Student implements Person {
      ...
    }

  • 设置组件扫描的基础包。我们之前说过，构造 Config 类可以让其同包下的 Bean 被 Spring 发现，但是有时候我们想将 Config 类和真正的业务类分开，这是就可以使用注解指定扫描基础包，语法为：

    xxxxxxxxxx
    @Configuration
    @ComponentScan("xyz.xijinian")
    public class PersonConfig {
      ...
    }

  • 超脱的 Autowired。使用 Autowired 的最常见形式是在类的开头把所需要的 Bean 注入进来，但事实上，Autowired 可以出现在构造器、Setter 方法、各种各样的方法上，Spring 会尝试满足方法参数上所声明的依赖，如：

    xxxxxxxxxx
    @Autowired
    public void insertPerson(Person person) {
      this.person = person;
    }

  • Java 依赖注入规范

    • @Named ≈ @Component
    • @Inject ≈ @Autowired

在 Java 中进行显示配置

含义

所谓的显示配置就是在 JavaConfig 类中不启用 @ComponentScan 注解的时候（依然有 @Configuration 注解），我们可以直接在 JavaConfig 类中使用 @Bean 注解来显示声明 Bean。

@Component 是写给类的，然后当类在"基础包"下会被自动扫描，直接被 Spring 装配为 Spring Bean。 @Bean 是写在 JavaConfig 类中的，你需要显式给出创建 Bean 的方法，所以说 @Bean 注解是手动装配。

简单的 Bean

在一个方法上面使用 @Bean 注解的时候，@Bean 注解会告诉 Spring 这个方法将会返回一个对象，该对象要注册为 Spring 应用上下文中的 Bean，方法体包含了最终产生 Bean 实例的逻辑：

xxxxxxxxxx
@Bean
public CompactDisc sgtPeppers() {
  return new sgtPeppers();
}

默认情况下，Bean 的 ID 与这个方法名字是一样的，你可以这样来起一个特立独行的名字：

xxxxxxxxxx
@Bean(name="lalala")
...

借助 java Config 来实现注入

xxxxxxxxxx
@Bean
public CDPlayer cdPlayer() {
  return new CDPlayer(sptPeppers());
}

这里有趣的地方在于，我们在构造 CDPlayer 时，使用了一个"函数"，而此函数我们曾将其声明为 Bean，于是 Spring 会拦截对方法的调用，直接返回 Bean 的实例。

事实上，上面所说的这些代码并不是特别好理解，我们有更优雅的办法：

xxxxxxxxxx
@Bean
public CDPlayer cdPlayer(CompactDisc compactDisc) {
  return new CDPlayer(compactDisc);
}

通过这种方式也会自动装配一个 CompactDisc 实例到配置方法中，而且这里甚至不要求将 CompactDisc 声明在 Java Config 里面，它可以来源于其他配置类、XML文件或者是自动装配出来的。不管 CompactDisc 采用什么方式创建，Spring 都会将其传入到配置方法中，并用来创建 CDPlayer Bean。

环境与 profile

类级别

@Profile 注解可以应用在类级别上。它会告诉 Spring 这个配置类中的 bean 只有在响应的 profile 激活时才会创建。

方法级别

在 Spring 3.1 中，只能在类级别上使用 @Profile 注解。不过，从 Spring 3.2 开始，也可以在方法级别上使用 @Profile 注解，与 @Bean 注解一同使用。这样的话，就能将这两个 bean 的声明放到同一个配置类之中：

xxxxxxxxxx
@Configuration
public class DataSourceConfig {
  
  @Bean(destroyMethod = "shutdown")
  @Profile("dev")
  public DataSource embeddedDataSource() {
    return new EmbeddedDatabaseBuilder()
        .setType(EmbeddedDatabaseType.H2)
        .addScript("classpath:schema.sql")
        .addScript("classpath:test-data.sql")
        .build();
  }
​

  @Profile("prod")
  public DataSource jndiDataSource() {
    JndiObjectFactoryBean jndiObjectFactoryBean = new JndiObjectFactoryBean();

    jndiObjectFactoryBean.setResourceRef(true);
    jndiObjectFactoryBean.setProxyInterface(javax.sql.DataSource.class);

激活 profile

Spring 在确定哪个 profile 处于激活状态时，需要依赖两个独立的属性：spring.profiles.active 和 spring.profiles.default 。 如 果 设 置 了 spring.profiles.active 属性的话，那么它的值就会用来确定哪个 profile 是激活的。但如果没有设置 spring.profiles.active 属性的话，那 Spring 将会查找 spring.profiles.default 的 值 。 如 果 spring.profiles.active 和 spring.profiles.default 均没有设置的话，那就没有激活的 profile，因此只会创建那些没有定义在 profile 中的 bean。有多种方式来设置这两个属性：

  • 作为 DispatcherServlet 的初始化参数；
  • 作为 Web 应用的上下文参数；
  • 作为 JNDI 条目；
  • 作为环境变量；
  • 作为 JVM 的系统属性；
  • 在集成测试类上，使用 @ActiveProfiles 注解设置。

Condition 接口

在 @Condition 后面的必然是一个实现了 Condition 接口的类，Condition 接口定义如下：

xxxxxxxxxx
public interface Condition {
  boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata);
}

上面接口中的参数也是两个接口：

  • ConditionContext

    定义如下：

    xxxxxxxxxx
    public interface ConditionContext {
      BeanDefinitionRegistry getRegistry();             // 检查 bean 定义
      ConfigurableListableBeanFactory getBeanFactory(); // 检查 bean 是否存在，甚至探查 bean 的属性
      Environment getEnvironment();                     // 检查环境变量是否存在以及它的值是什么
      ResourceLoader getResourceLoader();               // 读取并探查 ResourceLoader 所加载的资源
      ClassLoader getClassLoader();                     // 借助 ClassLoader 加载并检查类是否存在
    }

  • AnnotatedTypeMetadata

    定义如下：

    xxxxxxxxxx
    public interface AnnotatedTypeMetadata {
      boolean isAnnotated(String annotationType); // 判断带有 @Bean 注解的方法是不是还有其他特定的注解
      
      // 检查 @Bean 注解的方法上其他注解的属性

      Map<String, Object> getAnnotationAttributes(String annotationType, boolean classValuesAsString);

  • 官方示例 — 在 Spring 4.x 重新实现了 @Profile 注解：

    @Retention(RetentionPolicy.RUNTIME)

    class ProfileCondition implements Condition {
      public boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
        if (context.getEnvironment() != null) {
          MultiValueMap<String, Object> attrs =
              metadata.getAllAnnotationAttributes(Profile.class.getName());
          if (attrs != null) {

Bean 的作用域

四种作用域

Spring 定义了多种作用域，可以基于这些作用域创建 bean，包括：

  • 单例(Singleton)：在整个应用中，只创建 bean 的一个实例（Spring 默认）。
  • 原型(Prototype)：每次注入或者通过 Spring 应用上下文获取的时候，都会创建一个新的 bean 实例。
  • 会话(Session)：在 Web 应用中，为每个会话创建一个 bean 实例。
  • 请求(Rquest)：在 Web 应用中，为每个请求创建一个 bean 实例。

因为 StoreService 是一个单例的 bean，会在 Spring 应用上下文加载的时候创建。
当它创建的时候，Spring 会试图将 ShoppingCart bean 注入到 setShoppingCart() 方法中。
但是 ShoppingCart bean 是会话作用域的，此时并不存在。
直到某个用户进入系统，创建了会话之后，才会出现 ShoppingCart 实例。
而且系统中同时会存在很多 ShoppingCart 实例(一个用户一个)。
​
我们希望的是当 StoreService 处理购物车功能时，它所使用的 ShoppingCart 实例恰好是当前会话所对应的那一个。
Spring 并不会将实际的 ShoppingCart bean 注入到 StoreService，而会注入一个到 ShoppingCart bean 的代理。
这个代理会暴露与 ShoppingCart 相同的方法，所以 StoreService 会认为它就是一个购物车。
当调用 ShoppingCart 的方法时，代理会对其进行懒解析并将调用委托给会话作用域内真正的 ShoppingCart bean。
​
综上：
proxyMode 属性被设置成了 ScopedProxyMode.INTERFACES，这表明这个代理要实现 ShoppingCart 接口，并将调用委托给实现 bean。
​
另：
如果 ShoppingCart 是接口而不是类的话，这是可以的(也是最为理想的代理模式)。
但如果 ShoppingCart 是一个具体的类的话，Spring 就没有办法创建基于接口的代理了。
此时，它必须使用 CGLib 来生成基于类的代理。
所以，如果 bean 类型是具体类的话，我们必须要将 proxyMode 属性设置为 ScopedProxyMode.TARGET_CLASS，以此来表明要以生成目标类扩展的方式创建代理。
**/
