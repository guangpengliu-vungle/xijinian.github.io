基本概念和常识

  • POM 头

旧时代配置与自动配置

我们之前想将 Bean 注入 Spring 容器，要么就主动声明为 @Bean，要么就声明一个 @Component，然后等着被自动扫描（xml 方式就不特意说了，实际上和注解方式是差不多的）。

Spring Boot 带来了神奇的自动配置，它会在运行时检查类路径下的依赖，如果有某些相应的 jar 包（需要 Spring Boot 本来就认识，小众 jar 包是没这个待遇的），Spring Boot 会自动在 Spring 上下文中创建对应的 Bean。

这样带来了一个问题：我们之前手动创建 Bean 的时候可能会给 Bean 配置各种属性，比如 DataSource 的用户名、密码、URI… 那自动创建的 Bean 总不能直接得到我们心中所想的属性吧，确实如此，但 Spring Boot 会抽取多个属性源，通过他们来配置那些 Bean。

Spring Security IN ACTION

发表于 2018-06-03 Disqus：

启用 Spring Security

原理解读

Spring Security 借助一系列的 Servlet Filter 来提供各种安全性功能。

在使用 Spring 开发的时候，如果想让 Filter 享受到依赖注入的好处，常见的操作是实现 GenericFilterBean 并注册到 IoC 容器，为了启动它们，一般要在配置中声明一个 DelegatingFilterProxy，使其 filter-name 和 Ioc 中的 Filter 挂钩。

Spring Security 类似，它声明一个名为 springSecurityFilterChain 的 DelegatingFilterProxy，然后指向 IoC 容器中注册的一个 FilterChainProxy 的实例，FilterChainProxy 通过拓展 GenericFilterBean 间接实现了 Filter 接口，同时持有一组 SecurityFilterChain，

启用办法

在 Servlet 3.0 环境中，容器会在类路径中查找实现 javax.servlet.ServletContainerInitializer 接口的类，如有，就用它来配置 Servlet 容器。 Spring 提供了名为 SpringServletContainerInitializer 的实现，这个类反过来又会查找实现了 WebApplicationInitializer 的类并将配置的任务交给它们来完成。而 AbstractSecurityWebApplicationInitializer 就是 WebApplicationInitializer 的一个基础实现。
