使用 ResultSet 来卷动、更新数据

从 JDBC 2.0 开始，在使用 Connection 的 createStatement() 和 prepareStatement() 的时候，可以指定结果集类型和并行方式。

结果集类型可以指定三种设置：

  • ResultSet.TYPE_FORWARD_ONLY（默认）→ 只能前进光标，next()
  • ResultSet.TYPE_SCROLL-INSENSITIVE → 可以前后移动光标，不会反应数据库中的数据修改
  • ResultSet.TYPE_SCROLL-SENSITIVE → 可以前后移动光标，会反应数据库中的数据修改

并发方式有两种指定：

  • ResultSet.CONCUR_READ_ONLY（默认）→ 只能用 ResultSet 进行数据读取
  • ResultSet.CONCUR_UPDATABLE → 还可以对数据进行更新

批次更新

可以使用 addBatch() 方法来收集 SQL，并使用 executeBatch() 方法将所收集的 SQL 传送出去。

使用 executeBatch() 时，SQL 执行顺序就是 addBatch() 的顺序，最终会返回 int[]，代表每笔 SQL 造成的数据异动列表。执行 executeBatch() 时，先前已打开的 ResultSet 会被关闭，执行过后收集 SQL 用的 List 会被清空。

注意：① 执行批次更新的时候，要搭配关闭自动提交，更新完再打开自动提交。② 以 MySQL 为例，要支持批次更新，必须在 JDBC URL 上附加 rewriteBatchedStatements=true

事务

事务的四个基本要求是原子性（Atomicity）、一致性（Consistency）、隔离行为（Isolation behavior）和持续性（Durability），简称 ACID。

  • 原子性：一个事务有很多步骤，这些步骤要么都执行，要么都不执行。
  • 一致性：事务作用的数据集合在事务前后必须一致，不能使整个数据集合部分有变更，部分没变更。
  • 隔离行为：在多人使用的环境下，事务与事务必须互不干扰，用户不会意识到别的用户在进行事务。
  • 持续性：事务一旦变更，所有变更必须保存下来，即使系统发生了故障，事务的结果也不能遗失。

数据库的隔离级别

JDBC 可以通过 Connection 的 getTransactionIsolation() 取得数据库当前的隔离级别设置，可以通过 setTransactionIsolation() 来设置数据库的隔离级别。可设置常数是定义在 Connection 上的：

  • TRANSACTION_NONE
  • TRANSACTION_UNCOMMITTED → A 事务更新但未提交的数据，B 事务可读不可改 → 解决更新遗失
  • TRANSACTION_COMMITTED → A 事务更新并提交的数据，B 事务才可以读 → 解决脏读
  • TRANSACTION_REPEATABLE_READ → A 读取事务在提交前，B 事务可读不可改 → 解决不可重读读
  • TRANSACTION_SERIALIZABLE → 事务一个一个循序进行 → 解决幻读

事务传播行为

前面说过了事务隔离级别，这里再说说事务传播行为，当一个具有事务控制的方法被另一个有事务控制的方法调用后，需要如何管理事务（新建事务？在事务中执行？事务挂起？报异常？）

  • REQUIRED（默认值），在当前的事务中执行，如果当前没有事务，就新建一个事务。
  • SUPPORTS，在当前的事务中执行，如果当前没有事务，就在非事务状态下执行。
  • MANDATORY，必须在事务内部执行，如果当前有事务，就在事务中执行，如果没有事务，报错。
  • REQUIRES_NEW，必须在事务中执行，如果当前没有事务，新建事务，如果当前有事务，把当前事务挂起，新建事务。
  • NOT_SUPPORTED，必须在非事务下执行，如果当前没有事务，正常执行，如果当前有事务，把当前事务挂起。
  • NEVER，必须在非事务状态下执行，如果当前没有事务，正常执行，如果当前有事务，报错。
  • NESTED：必须在事务状态下执行，如果没有事务，新建事务，如果当前有事务，创建一个嵌套事务。实际上使用了保存点技术。

我们常用的三个已经在上面加粗标记。

@Transactional 失效问题

如果我们在一个类中自调用了 @Transactional 注解方法，这个注解就会失效，理由很简单，因为 Spring 事务控制基于 AOP，而 AOP 基于动态代理，自调用没法使用动态代理。可以通过 getBean 之类的办法，手动调用代理。

错误使用 ServiceImpl

在一个 Service 方法中连续调用了两个带事务的方法，并不能保证两个事务都能执行成功。

不要滥用事务

事务代码中应该都和事务有关，无关的代码拿到事务注解外，否则数据库的连接数撑不住。

错误捕捉异常

在事务中 catch 住异常后，可以做一些善后工作，但之后还是要抛出一个异常，因为希望让 Spring 进行事务管理，如果你手动把异常吞了，Spring 就看不到，也不会回滚了。

接口说明

我们来看看我们写的接口格式：

  1. 查询动词

    Spring Data 允许四种查询动词，如 get、read、find 和 count，其中前三个是同义词。

  2. 主题

    对于大部分场景来说，主题会被省略掉，因为要查询的对象类型是通过如何参数化 JpaRepository 接口来确定的，而不是方法名称中的主题。有一种例外情况，如果主题的名称以 Distinct 开头的话，那么在生成查询的时候会确保所返回结果集中不包含重复记录。

  3. 断言

    断言中会有一个或多个限制结果的条件，每个条件必须引用一个属性，并且还可以指定一种比较操作。如果省略比较操作符的话，那么这暗指是一种相等比较操作，其他比较操作如下：

    • IsAfter、After、IsGreaterThan、GreaterThan
    • IsGreaterThanEqual、GreaterThanEqual
    • IsBefore、Before、IsLessThan、LessThan
    • IsLessThanEqual、LessThanEqual
    • IsBetween、Between
    • IsNull、Null、IsNotNull、NotNull
    • IsIn、In、IsNotIn、NotIn
    • IsStartingWith、StartingWith、StartsWith
    • IsEndingWith、EndingWith、EndsWith
    • IsContaining、Containing、Contains
    • IsLike、Like
    • IsNotLike、NotLike
    • IsTrue、True
    • IsFalse、False
    • Is、Equals
    • IsNot、Not
