REST 基础知识

REST 和 RPC 关系不大：RPC 是面向服务的，并关注行为和动作；REST 是面向资源的。

RE[$Representational^{表述性}$]S[$State^{状态}$]T[$Transfer^{转移}$] —— 就是将资源状态以最适合客户端或服务端的形式从服务器端转移到客户端（或者反过来）。

在 REST 中，资源通过 URL 进行识别和定位，关注的是资源，而不是行为。REST 中会有行为，但不体现在 URL 中，而是通过 HTTP 方法定义的，和 CRUD 对应：

  • $POST^{Create}$
  • $GET^{Read}$
  • $PUT/PATCH^{Update}$
  • $DELETE^{Delete}$

Spring REST 综述

  • 控制器可以处理所有的 HTTP 方法
  • 借助 @PathVariable 注解，控制器能够处理参数化的 URL（将变量作为 URL 的一部分）
  • 借助视图解析器，资源能够以多种方式表达：JSON、XML、RSS…
  • 可以使用 ContentNegotiatingViewResolver 来选择适合客户端的表述
  • 借助 @ResponseBody 注解和 HttpMethodConverter 实现，可以替换基于视图的渲染方式
  • 借助 @RequestBody 注解和 HttpMethodConverter 实现可以将传入的 HTTP 数据转化为 Java 对象
  • 借助 RestTemplate，Spring 可以方便地调用 REST 资源

转换资源形式

控制器以 Java 对象的方式来处理资源，它本身通常不关心资源如何表达。控制器完成了工作后，资源才会被转化成最适合客户端的形式。

ContentNegotiatingViewResolver 特殊的地方在于它可以：① 确定请求的媒体类型；② 找到适合请求媒体类型的最佳视图

确定请求的媒体类型

ContentNegotiatingViewResolver 首先考虑 URL 文件的拓展名，如果文件拓展名不能得到任何媒体类型，接着考虑到 Accept 头部信息，如果还是没有，ContentNegotiatingViewResolver 将会使用“/”作为默认的内容类型，这就意味着客户端必须要接收服务器发送的任何形式的表述。

确定了请求的媒体类型之后，ContentNegotiatingViewResolver 本身不会解析视图，而是委托给其他的视图解析器，解析得到的每个视图都会放在一个列表中，这个列表装配完成后，ContentNegotiatingViewResolver 会循环客户端请求的所有媒体类型，在候选的视图中查找能够产生对应内容类型的视图，第一个匹配的视图会用来渲染模型。

影响媒体类型的选择

上面描述了确定所请求媒体类型的默认策略。但是通过为 ContentNegotiatingViewResolver 注入一个 ContentNegotiationManager，我们能够改变它的行为。比如：

  • 指定默认的内容类型
  • 通过请求的参数指定内容的类型
  • 忽视请求的 Accept 头部信息
  • 将请求的扩展名映射为特定的媒体类型

配置 ContentNegotiationManager

  • 直接声明一个 ContentNegotiationManager 类型的 bean

  • 通过 ContentNegotiationManagerFactory 间接创建 bean（XML 推荐这么配置）

  • 重载 WebMvcConfigurerAdapter 的 configureContentNegotiation() 方法（Java 注解驱动推荐这个）

以前我们配置很多行为都是在 ContentNegotiatingViewResolver 中配置的，新版本的 Spring 鼓励使用 ContentNegotiationManager，甚至为此直接把很多 ContentNegotiatingViewResolver 中的 setter 方法废弃了。

优势与劣势

  • 它在 Spring MVC 之上构建了 REST 资源表述层，控制器代码无需修改。相同的一套控制器方法能够为面向人类的用户产生 HTML 内容，也能针对不是人类的客户端产生 JSON 或 XML。然而如果面向人类用户的接口与面向非人类客户端的接口之间没有很多重叠的话，它的最大优势就不在了。

  • ContentNegotiatingViewResolver 作为 ViewResolver 的实现，它只能决定资源该如何渲染到客户端，而不能建议客户端要发送什么样的表述给控制器使用。

  • ContentNegotiatingViewResolver 所选中的 View 会渲染模型给客户端，而不是资源。

消息转换器

在响应体中返回资源状态

正常情况下，当处理方法返回 Java 对象（除 String 外或 View 的实现以外）时，这个对象会放在模型中并在视图中渲染使用。如果使用了消息转换功能的话，我们需要告诉 Spring 跳过正常的模型/视图流程，并使用消息转换器。最简单的方法是为控制器方法添加 @ResponseBody 注解。

@ResponseBody 注解会告知 Spring，我们要将返回的对象作为资源发送给客户端，并将其转换为客户端可接受的表述形式。更具体地讲，DispatcherServlet 将会考虑到请求中 Accept 头部信息，并查找能够为客户端提供所需表述形式的消息转换器。

在默认情况下，Jackson 库在将返回的对象转换为 JSON 资源表述时，会使用反射。我们可以在 Java 类型上使用 Jackson 的映射注解，从而改变产生 JSON 的行为。

为控制器默认设置消息转换

Spring 4.0 引入了 @RestController 注解，如果在控制器类上使用 @RestController，Spring 将会为该控制器的所有处理方法应用消息转换功能。我们不必为每个方法都添加 @ResponseBody 了。

RestTemplate 还提供了 getForEntity()。getForEntity() 与 getForObject() 相似。getForObject() 只返回资源（通过 HTTP 信息转换器将其转换为 Java 对象），getForEntity() 会在 ResponseEntity 中返回相同的对象，同时还带有关于响应的额外信息，如 HTTP 状态码和响应头。

PUT

对象将被转换成什么样的内容类型很大程度上取决于 put() 的参数类型：

  • 如果参数是 String，那么将会使用 StringHttpMessageConverter 将这个值直接被写到请求体中并设置内容类型为 “text/plain”；
  • 如果给定一个 MultiValueMap<String, String>，那么这个 Map 中的值将会被 FormHttpMessageConverter 以 “application/x-wwwform- urlencoded” 的格式写到请求体中；
  • 所以参数类型是 Java 对象，且在类路径下包含 Jackson 2 库，那么 MappingJacksonHttpMessageConverter 将以 “application/json” 格式将 Java 对象 写到请求中。
