ElasticSearch 入门

索引、文档和 REST API

文档

文档是所有可搜索数据的最小单位，类比 MySQL 就相当于是表中的一行数据。文档会被序列化为 JSON 保存。每个文档都有一个 Unique ID，你可以指定或者交给 Elasticsearch 自动生成。

每一个文档都有元数据，如：

  • _index
  • _type：7.0 后只能设置一个 Type，即 _doc
  • _id：Unique ID
  • _source：原始 JSON 数据
  • _all：7.x 中已被废除
  • _version
  • _score：相关性打分

索引

索引是文档的容器，是一类文档的组合，类似 MySQL 中的表，里面有很多行（文档）。

每个索引都有自己的 Mapping 定义，用于包含文档的字段名和字段类型，类似于 MySQL 的表结构。

每个索引都有自己的 Setting 定义，定义数据如何分布，有多少个分片。

索引中的数据分布在 Shard 分片上。

节点、集群、分片及副本

节点

本质上是一个 Java 进程，一个机器上可以运行多个节点，但是生产环境一般一台机器上只有一个。

每一个节点都有名字，同步配置文件配置，或者启动的时候通过 -E node.name=node1 来指定。

每一个节点在启动后，都会分配一个 UID，保存在 data 目录下。

Master-eligible Node 和 Master Node

每个节点启动以后，默认就是一个 Master eligible 节点，可以设置 node.master:false 禁止。

Master-eligible 节点可以参加选主流程，成为 Master 节点。当第一个节点启动的时候，它就会把自己选举成 Master 节点。

集群状态（Cluster State）是指一个集群中：① 所有的节点信息；② 所有的索引和其相关的 Mapping、Setting 信息；③ 分片的路由信息。

每一个节点上都保存了集群状态，但是只有 Master 节点才能修改集群状态（为了防止造成数据的不一致）。

Data Node 和 Coordinating Node

Data Node 就是可以保存数据的节点，负责保存分片的数据，在数据扩展上起到了重要的作用。

Coordinating Node 负责接收 Client 的请求，将请求分发到合适的 Node，最终把结果混在一起。每个节点默认都起到了 Coordinating Node 的职责。

其他类型的节点

Hot & Warm Node

Machine Learning Node

Tribe Node（5.3 开始使用 Cross Cluster Search），它连接到不同 Elasticsearch 集群，并且支持将这些集群当成一个单独的集群处理。

配置节点类型

每个节点在启动的时候会读取 elasticsearch.yml 配置文件来设置自己的角色，生产环境中最好设置单一角色，开发环境中一个节点可以承担多种角色。

节点类型 配置参数 默认值
master eligible node.master true
data node.data true
ingest node.ingest true
coordinating only 默认都是 coordinating 节点
设置其他类型全部为 false
machine learning node.ml true（需要 enable x-pack）

分片

  1. 主分片，可以将数据分布到集群内的所有节点之上

    一个分片是一个运行的 Lucene 的实例

    主分片数在索引创建时指定，后续不允许修改，除非 Reindex

  2. 副本，解决数据高可用，是主分片的拷贝

    副本分片数可以动态调整

    增加副本数，还可以在一定程度上提高服务的可用性（读取的吞吐）

分片的设定

  • 分片数设置过小

    • 就算以后节点增加了，分片数也改不了，无法通过增加节点来实现水平扩展

    • 单个分片的数据量太大，数据重新分配耗时

  • 分片数设置过大（7.0 开始，默认分片数从 5 改成了 1，因为数据量不大，如几百万，一个主分片就够了）

    • 影响搜索结果的相关性打分，影响统计结果的准确性
    • 单个节点上过多的分片，徒增消耗
